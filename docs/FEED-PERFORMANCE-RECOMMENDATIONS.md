# 피드·첫 진입 로딩 속도 개선 권장사항

로그인/온보딩 후 첫 피드 진입 시 2~4초 이상 걸리는 현상을 줄이기 위한 벤치마킹 요약과, **기능을 침범하지 않는** 개선안을 정리했다.  
또한 **데이터가 방대해졌을 때**(인스타/핀터레스트 수준)에도 “들어올 때마다 새롭고 맞춤형인 피드”를 유지하면서 **메인 피드가 너무 오래 걸리지 않도록** 하는 균형 전략을 담았다.

---

## 0. 설계 원칙: 속도와 새로움의 균형

### 0.1 목표

- **첫 화면은 빠르게**: 메인 피드(작품·전시)가 1~2초 안에 그려져야 한다. 로딩 스피너만 길게 보이면 이탈이 늘어난다.
- **방문마다 새로움**: 데이터가 많아지면 매번 같은 50개가 아니라, **선호에 맞으면서도 새로 올라온/다양한** 작품·전시가 섞여 들어와야 한다. 단순 캐시·고정 순서만으로는 “매번 비슷한 피드”가 된다.
- **로직은 단순하지 않게, 하지만 첫 로딩은 가벼우면**: 추천·랭킹·새로고침 정책은 나중에 더 정교해질 수 있지만, **첫 요청이 “한 번에 모든 걸 기다리는” 구조**는 피해야 한다.

### 0.2 전략 요약

| 구분 | 전략 | 이유 |
|------|------|------|
| **첫 페이지만 먼저** | 첫 로딩 시 “첫 페이지”(예: 12~24개)만 요청하고, 나머지는 “더 보기”/무한스크롤 | 데이터가 많아져도 LCP가 늘어나지 않음. 백엔드가 커서/시간 구간으로 **매번 다른 슬라이스**를 줄 수 있어 “새로움” 확보 가능 |
| **단계적 로딩(Phased)** | 1단계: 작품+전시만 빠르게 → 화면에 바로 표시 / 2단계: 좋아요 상태·추천(discovery) 블록은 병렬로 이어서 받아서 합치기 | 첫 페인트는 “콘텐츠 유무”에만 의존하고, 개인화는 블로킹하지 않음. 매 방문마다 2단계에서 **최신 추천**을 받으므로 단순 고정 캐시가 아님 |
| **중복 fetch 제거** | userId 없을 때 1차 fetch 스킵, discovery 4회 순차 → 1회 병렬 | 같은 데이터를 두 번 끌어오거나 불필요한 직렬 RTT 제거. “새로움”은 **요청 횟수 감소**와 상충하지 않음 |
| **스켈레톤** | 데이터 도착 전 그리드 형태 스켈레톤 표시 | 체감 대기 시간 감소. 데이터 양과 무관하게 “화면이 채워진다”는 인상 |
| **캐시** | 피드 메인 리스트는 **긴 TTL 캐시 금지**. (선택) 재방문 시 잠깐 “이전 결과” 보여주고 백그라운드 갱신(stale-while-revalidate)은 가능 | 매 방문마다 새 데이터를 볼 수 있게. SWR은 “즉시 보여주기 + 곧 새로 고침”이라 새로움과 속도 둘 다 만족 |

### 0.3 데이터가 많아졌을 때 (확장 시)

- **백엔드**: “최신 50개 고정” 대신 **커서·시간 구간·랜덤 시드·랭킹** 등을 넣어 “첫 페이지”를 구성하면, 방문할 때마다 다른 슬라이스가 나올 수 있다. 한 번에 50+30을 주는 대신 “첫 12~24개” RPC/API 하나로 묶어도 됨.
- **클라이언트**: 첫 로딩은 “첫 페이지” 한 번 + (선택) 좋아요/팔로우 상태만 추가로; discovery는 2단계에서 항상 **이번 세션 기준**으로 받아서 넣으면, 단순해 보이면서도 매번 새 추천이 섞인다.

### 0.4 피드 개인화 방향: 지금 vs 리치해졌을 때

**지금 (유저·액션·데이터가 적을 때)**  
- 메인 피드(전체 탭)는 **전역 최신/인기 순** — 누가 들어와도 같은 50작품·30전시를 보게 될 확률이 높다.  
- 이 단계에서는 괜찮다. Discovery 블록만 유저별(좋아요·팔로우 기반)이고, 나머지는 “공통 진입 화면”으로 두어도 된다.

**리치해졌을 때 (유저·데이터·행위가 쌓인 뒤)**  
- 유저 수, 작품/전시 수, **행위 데이터**(좋아요, 팔로우, 가격 문의, 조회 등)가 늘어나면 **유저마다 다른 메인 피드**가 필요해진다.  
- “같은 전역 50개”가 아니라, **그 유저에게 맞는** 작품·전시가 메인에 노출되어야 한다. 즉, **로딩 퍼포먼스**와 **커스터마이제이션**을 **둘 다** 신경 쓰고 균형을 맞춰야 한다.

**개인화에 쓸 수 있는 신호 (최소·점진적)**  
- **온보딩/프로필**: 역할(아티스트·큐레이터·갤러리·컬렉터), 관심 매체·테마, 지역 등.  
- **위치**(선택): 지역 기반 큐레이션·전시 노출.  
- **행위**: 좋아요, 팔로우, 가격 문의, 작품/프로필 조회, 전시 참여 등.  
→ 이걸 활용해 백엔드에서 “첫 페이지”를 **유저별로 다른 순서·다른 구성**으로 만들면, 리치해졌을 때 **완전히 다른 피드 UX**를 유저마다 줄 수 있다.

**퍼포먼스와의 균형**  
- 개인화 로직(랭킹·필터·다양성)이 무거워지면, “한 번에 모든 걸 계산”하면 로딩이 느려진다.  
- 그래서 **단계적 로딩**(Phase1에서 이미 “유저별 첫 페이지” 1회 호출) + **첫 페이지만** + 필요 시 **백엔드 캐시/프리컴퓨트**를 두면, **빠른 첫 화면**과 **유저별 메인 피드**를 동시에 만족시킬 수 있다.

이하 섹션 1~4는 **지금 코드 기준**의 병목과 즉시 적용 가능한 개선안이다. 위 원칙을 지키면 나중에 “전역 피드 → 유저별 메인 피드”로 전환할 때도, 로딩 퍼포먼스와 커스터마이제이션 균형을 유지할 수 있다.

## 1. 현재 로드 흐름 요약

### 1.1 진입 경로

- **`/`** → `getSession()` → `getMyProfile()` → (온보딩/비밀번호 체크) → **`/feed?tab=all&sort=latest`** 리다이렉트
- **`/feed`** 직접 접근 시: `AuthGate`(session + profile) 통과 후 `FeedClient` → `FeedContent`

### 1.2 블로킹·중복 호출

| 단계 | 위치 | 동작 | 비고 |
|------|------|------|------|
| 1 | AuthGate | `getSession()` → `getMyProfile()` | 직렬 2회, 통과 전까지 children 미렌더 |
| 2 | Header | `getSession()` → `getMyProfile()` + `getUnreadCount()` | session/profile 재조회 |
| 3 | FeedClient | `getSession()` → `setUserId()` | session 3번째 조회, **userId 설정 전까지 FeedContent는 userId=null** |
| 4 | FeedContent | `fetchArtworks()` (userId=null) | 1차 풀 fetch (아트워크·전시·following·liked, **discovery 없음**) |
| 5 | FeedContent | userId 갱신 후 `fetchArtworks()` 재실행 | **2차 풀 fetch** (동일 데이터 + recs + discovery) |

→ **같은 피드 데이터를 두 번 풀 로드**하고, **getSession / getMyProfile**이 여러 컴포넌트에서 중복 호출된다.

### 1.3 FeedContent 내부 워터폴

`fetchArtworks()` 한 번의 대략적 순서:

1. **병렬 1**: `listPublicArtworks(50)` or `listFollowingArtworks(50)` + `getFollowingIds()` + `listPublicExhibitionsForFeed(30)` or `listExhibitionsForFeed(...)`
2. **직렬**: `getLikedArtworkIds(allIds)` (1 RTT)
3. **직렬**: `fetchRecProfiles()` = `getPeopleRecs(likes_based)` + `getPeopleRecs(follow_graph)` (2 RPC 병렬)
4. **직렬 루프**: `for (p of recProfiles.slice(0, 4))` → **`listPublicArtworksForProfile(p.id)` × 4회 순차** (각 프로필당 내부 2쿼리 → 총 8 RTT 직렬에 가깝게 동작)

즉, **discovery 블록 4개를 위해 최대 4번의 순차 네트워크 라운드**가 추가된다.

### 1.4 기타 무거운 페이지 후보

- **People** (`/people`): 추천 3레인 + 검색 시 변형 쿼리·suggestion
- **My** (`/my`): KPI(팔로잉/팔로워/작품/문의/클레임) + 탭별 데이터
- **전시 상세** (`/e/[id]`): 전시 메타 + 작품 목록 + 참여 작가

이 문서에서는 **피드 첫 로딩**에 집중하고, 동일 원칙을 다른 페이지에 적용할 수 있다.

## 2. 권장 개선안 (기능 유지 전제)

아래 개선안은 **속도**와 **방문마다 새로움**을 동시에 만족하도록 설계했다. 단순히 “캐시만 키우자”가 아니라, **첫 페인트는 최소 요청으로**, **개인화·추천은 블로킹하지 않고 이어서** 넣는 방식이다.

---

### 2.0 [핵심] 단계적 로딩(Phased Loading) — 빠른 첫 화면 + 매번 새 추천

**목표**: 메인 피드가 빨리 뜨고, 좋아요/추천(discovery)은 그 위에 “이번 방문 기준”으로 자연스럽게 합쳐진다.

**Phase 1 (블로킹, 최소한만)**  
- **요청**: 작품 목록(첫 페이지 한도, 예: 18~24개) + 전시(첫 페이지 한도, 예: 6~12개) + (tab=following이면) getFollowingIds.  
- **표시**: 이 데이터만 오면 바로 그리드 렌더. 좋아요 하트는 기본값(빈 Set) 또는 스켈레톤 상태로 두어도 됨.  
- **특징**: RTT 1~2번으로 첫 화면 완성. 데이터가 커져도 “첫 페이지 한도”만 쓰면 LCP가 일정하게 유지됨.

**Phase 2 (논블로킹, 병렬)**  
- **요청**: (Phase 1 직후 또는 동시에)  
  - `getLikedArtworkIds(phase1ArtworkIds)`  
  - `getPeopleRecs(likes_based)` + `getPeopleRecs(follow_graph)` → `Promise.all`로 4명분 `listPublicArtworksForProfile`  
- **표시**: 각각 도착하는 대로 state 반영(likedIds 갱신, discovery 블록 삽입). 이미 그려진 그리드 위에 “하트 채워짐”·“추천 블록”만 업데이트.  
- **특징**: 매 방문마다 **이번 세션에서 받은 추천**을 쓰므로, 단순 고정 캐시가 아님. 인스타/핀터레스트처럼 “들어올 때마다 새 추천”이 섞인다.

**구현 포인트**  
- Phase 1이 끝나면 `setLoading(false)` 또는 “첫 배치만 로딩 끝”으로 두고, Phase 2는 별도 state(예: `discoveryLoaded`)로 관리해도 됨.  
- “더 보기”가 있으면 Phase 1은 “첫 페이지”만, Phase 2 이후에 “더 보기”로 이어지는 기존 로직과 공존 가능.

**기능 영향**: 없음. 보여주는 데이터 구성은 그대로, **순서만 “먼저 메인 → 이어서 좋아요·추천”**으로 바뀐다.

**첫 페이지만 로드 (확장 대비)**  
- 지금은 `limit: 50`(작품) + 30(전시)을 한 번에 받고 있다. 데이터가 방대해지면 이 양이 LCP를 늘리고, 매번 비슷한 “앞쪽 50개”만 보이게 될 수 있다.  
- **권장**: 첫 로딩 시에는 **“첫 페이지”만** 요청 (예: 작품 18~24, 전시 6~12). 나머지는 “더 보기”/무한스크롤로 이어받기.  
- 이렇게 하면 (1) 첫 화면이 항상 가볍고, (2) 백엔드에서 커서·시간 구간·랭킹을 넣어 “첫 페이지”를 바꿀 수 있어 **방문마다 다른 새 작품**을 줄 수 있다.  
- 즉시 적용할 필요는 없고, 데이터가 늘어나는 시점에 limit 분리 + 커서/더보기와 함께 도입해도 된다.

### 2.1 [높은 영향] FeedContent: userId 없을 때 1차 fetch 생략

**문제**: `userId`가 `null` → `string`으로 바뀔 때 `fetchArtworks()`가 **두 번** 돌아가며, 1차는 discovery 없이 풀 리스트만 가져오고 2차에서 같은 리스트 + recs + discovery를 다시 가져온다.

**권장**:

- `userId === null`이면 **아예 1차 fetch를 하지 않는다.**
- `FeedClient`에서 session을 받아 `userId`를 넘기기 전까지는 “로딩” 또는 **스켈레톤**만 보여 주고, **userId가 확정된 뒤 단 1번만** `fetchArtworks()` 실행.

**기능 영향**: 없음. 비로그인으로 피드를 보는 경로가 없으므로, “한 번만 fetch”로 통일해도 동작은 동일하다.

**예상 효과**: 첫 진입 시 **동일한 대용량 요청 1회 제거** → 네트워크/DB 부하 및 체감 시간 감소 (수백 ms ~ 1초 이상 가능).

---

### 2.2 [높은 영향] Discovery 블록: 순차 4회 → 1회 병렬

**문제**: `recProfiles.slice(0, 4)`에 대해 `listPublicArtworksForProfile(p.id)`를 **for 루프로 await** 해서 4번 순차 실행된다.

**권장**:

```ts
const discoveryPromises = recProfiles.slice(0, DISCOVERY_BLOCKS_MAX).map((p) =>
  listPublicArtworksForProfile(p.id, { limit: 3 }).then(({ data: arts }) =>
    (arts ?? []).length > 0 ? { profile: p, artworks: arts ?? [] } : null
  )
);
const results = await Promise.all(discoveryPromises);
const discoveryWithArtworks = results.filter((r): r is NonNullable<typeof r> => r != null);
```

**기능 영향**: 없음. 결과 구성만 “순차 4회” → “병렬 1회”로 바뀐다.

**예상 효과**: discovery 구간 **4 RTT → 1 RTT**로 축소 → 수백 ms 절감 가능.

---

### 2.3 [중간 영향] userId 공유로 FeedClient의 getSession 제거

**문제**: `AuthGate`가 이미 session을 검사하고, `FeedClient`는 **다시** `getSession()`으로 `userId`만 얻는다. 세션은 보통 메모리/로컬 스토리에서 오므로 비용이 크지 않지만, 불필요한 직렬·중복이다.

**권장**:

- **옵션 A**: `AuthGate`(또는 상위 레이아웃)에서 session/userId를 context로 내려주고, `FeedClient`는 context의 `userId`만 사용. Feed 전용 페이지에서는 `getSession()` 호출 제거.
- **옵션 B**: 최소한 Feed 페이지에서만, `AuthGate`가 “통과 시” session을 children에 prop/context로 넘기고, `FeedClient`는 그 session에서 `user.id`를 써서 **한 번만** `userId`를 설정.

**기능 영향**: 없음. 단일 소스에서 userId를 쓰도록 정리.

**예상 효과**: 약간의 JS/네트워크 중복 제거, 코드 일관성·유지보수성 향상.

---

### 2.4 [중간 영향] 첫 페인트 전에 “스켈레톤” 노출

**문제**: `loading === true`인 동안 “Loading...” 텍스트만 보이면, 실제 데이터가 오기까지 2~4초가 걸릴 때 체감 대기 시간이 길다.

**권장**:

- 피드 그리드와 동일한 레이아웃의 **스켈레톤 카드**를 6~12개 정도 표시 (카드형 placeholder).
- 데이터 도착 후 스켈레톤을 실제 카드로 한 번에 치환.

**기능 영향**: 없음. 표시만 바뀐다.

**예상 효과**: LCP·체감 속도 개선 (콘텐츠가 “바로 그려진다”는 인상).

---

### 2.5 [낮은 영향·선택] getLikedArtworkIds 병렬화

**문제**: `getLikedArtworkIds(allIds)`는 **artwork 목록을 받은 뒤** 한 번 호출되므로, 구조상 1 RTT가 직렬로 붙는다.

**권장**:

- “메인 피드 목록”과 “liked 여부”를 완전히 동시에 시작하려면, **artwork id 없이** “현재 유저가 좋아요한 artwork id 집합”만 먼저 가져오는 API가 있으면 가능하다.  
  지금 구조에서는 **유지**해도 무방하고, 2.1·2.2 적용만으로도 이 RTT 비중은 상대적으로 작아진다.
- 만약 `artwork_likes`에 “user_id + 최근 N개” 같은 경량 쿼리가 있다면, “첫 화면 20개만 liked 체크” 등으로 범위를 줄일 수는 있다 (구현 난이도 대비 효과는 중간).

---

### 2.6 [참고] Supabase cache: no-store

현재 `createGuardedFetch`에서 **모든 요청에 `cache: 'no-store'`** 를 적용하고 있다.  
피드·아트워크는 “최신 상태”가 중요하므로, 전역 no-store를 바꾸면 **캐시 오염·stale 노출** 위험이 있다.  
따라서 “첫 로딩 속도”를 위해 no-store를 제거하는 것은 **권장하지 않는다.**  
대신 **서버/엣지에서 짧은 revalidate** 또는 **클라이언트에서 한 번만 fetch(2.1)** 로 줄이는 쪽이 안전하다.

## 3. 적용 우선순위 요약

**균형 원칙**: “메인 피드가 빨리 뜨게” + “들어올 때마다 새롭고 맞춤형”을 함께 만족하려면, **단순 캐시/고정 리스트**보다 **단계적 로딩 + 중복 제거 + (확장 시) 첫 페이지만**이 맞다.

| 순위 | 항목 | 난이도 | 기대 효과 | 기능 영향 |
|------|------|--------|-----------|-----------|
| 0 | **단계적 로딩**: Phase1(작품+전시) 먼저 표시 → Phase2(좋아요+discovery) 병렬로 합치기 | 중간 | 첫 화면 빠름 + 매 방문 새 추천 유지 | 없음 |
| 1 | FeedContent: userId 없을 때 1차 fetch 스킵 | 낮음 | 높음 (중복 1회 제거) | 없음 |
| 2 | Discovery 4회 순차 → Promise.all 1회 | 낮음 | 높음 (4 RTT → 1 RTT) | 없음 |
| 3 | userId context/prop 공유로 Feed getSession 제거 | 중간 | 중간 | 없음 |
| 4 | 피드 스켈레톤 UI | 중간 | 체감 속도·LCP | 없음 |

## 4. 벤치마킹 방법 제안

- **Chrome DevTools → Network**: “Disable cache” 켠 상태에서 `/feed` 직접 로드 후,  
  `listPublicArtworks` / `listPublicExhibitionsForFeed` / `get_people_recs` / `listPublicArtworksForProfile` 등 **요청 개수·순서·총 시간** 확인.
- **Performance 패널**: LCP, “Loading...” → 실제 그리드가 그려지기까지 시간 측정.
- **개선 후**: 동일 조건에서 2.1·2.2 적용 전/후로 요청 수와 체감 로딩 시간을 비교하면, “첫 진입 속도” 개선 효과를 수치로 확인할 수 있다.

이 순서대로 2.1 → 2.2를 적용하면, 기존 기능을 유지한 채 피드 첫 로딩 시간을 줄이는 데 가장 큰 효과를 낼 수 있다.

## 5. 요약: 로딩 퍼포먼스와 커스터마이제이션 균형

- **지금**: 유저·액션·데이터가 적어서 모두 비슷한(전역 최신/인기) 피드를 보는 건 수용 가능. 리치해지면 유저마다 다른 메인 피드가 필요하고, 그때는 최소한 위치·온보딩 정보·행위(좋아요, 팔로우, 가격 문의 등)만으로도 완전히 다른 피드 UX를 줄 수 있어야 함.
- **균형**: **로딩 퍼포먼스**와 **커스터마이제이션**을 **둘 다** 신경 써야 한다. 로직을 너무 단순하게 가져가면 새로움/맞춤이 사라지고, 한 번에 모든 걸 기다리게 하면 첫 화면이 느려진다.

**이 문서에서 제안하는 균형**:

1. **첫 화면은 “최소 데이터”로만 그리기**  
   작품+전시 첫 배치만 받고 바로 렌더. 좋아요/추천(discovery)은 **그 다음** 병렬로 받아서 합치기(단계적 로딩).  
   → 메인 피드가 빨리 뜨고, 추천은 매 방문마다 **이번 세션** 기준으로 갱신됨.

2. **불필요한 중복·직렬 제거**  
   userId 없을 때 1차 fetch 스킵, discovery 4회 순차 → 1회 병렬.  
   → 새로움을 해치지 않으면서 RTT와 중복만 줄임.

3. **확장 시: 유저별 메인 피드 + 첫 페이지만**  
   데이터가 커지면 **메인 리스트 자체를 유저별**로(온보딩·위치·행위 기반 랭킹). 첫 로딩은 “첫 18~24개”만 요청하고 더보기로 이어받기. 백엔드가 유저 id를 받아 첫 페이지를 다르게 주면, 유저마다 다른 피드 UX를 로딩이 느려지지 않게 줄 수 있음.

4. **캐시**  
   피드 메인 리스트는 긴 TTL 캐시로 고정하지 않음. (선택) 재방문 시 잠깐 이전 결과 보여주고 백그라운드 갱신(SWR)은 가능.

이렇게 하면 “메인 피드가 뜨는데 너무 오래 걸리지 않으면서”, “데이터가 많아져도 들어올 때마다 (선호에 맞는) 새로운 작품들이 추천되는” 구조를 유지할 수 있다. **로딩 퍼포먼스**와 **커스터마이제이션**(리치해졌을 때 유저마다 다른 메인 피드)을 둘 다 신경 쓰고 밸런스를 맞추는 것이 목표다.
