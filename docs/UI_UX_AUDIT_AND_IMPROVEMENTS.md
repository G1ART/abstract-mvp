# UI/UX 전수 감사 및 개선 제안

최초 로그인·온보딩을 포함한 전체 페이지와 페이지 간 관계를 UI/UX 관점에서 점검한 결과와, 타 플랫폼 벤치마킹을 반영한 개선 방안(옵션별 순위)을 정리한 문서.

---

## 1. 페이지·플로우 개요

```
[비로그인]
  / (홈) → 로그인 없음 → /login
  /login → 비밀번호 로그인 | 매직링크 → (성공 시) / 또는 next
  /onboarding → signup(이메일·비밀번호·유저네임·역할) | 이미 세션 시 프로필 완성(매직링크 진입)
  /auth/callback → 매직링크/확인 후 → 프로필 없음 /onboarding | 비밀번호 없음 /set-password | 그 외 /feed 또는 next
  /invites/delegation?token=… → 로그인 유도 또는 수락

[로그인·온보딩 완료]
  / → 프로필 있음, 비밀번호 있음 → /feed
  /set-password → 비밀번호 설정(매직링크 유저) → /feed
  /feed (전체/팔로잉, 최신/인기)
  /people (검색·추천)
  /upload (단일 | 벌크 | 전시 만들기)
  /my (내 프로필·작품·전시·탭)
  /my/exhibitions, /my/exhibitions/new, /my/exhibitions/[id], /my/exhibitions/[id]/add, /my/exhibitions/[id]/edit
  /my/delegations, /my/claims, /my/inquiries, /my/followers, /my/following
  /settings
  /notifications
  /artwork/[id], /artwork/[id]/edit
  /u/[username] (공개 프로필)
  /e/[id] (전시 공개)
```

## 2. 진입·인증

### 2.1 홈(/) — 로딩만 표시 후 리다이렉트

**현상**  
- "Loading..." 한 줄만 보이고 곧바로 `/login` 또는 `/feed` 등으로 이동.  
- 서비스명·로고·한 줄 설명이 없어, 새 탭에서 `/`를 열었을 때 "뭔가 불러오는 중"으로만 보일 수 있음.

**벤치마크**  
- Notion, Linear, Figma: 로고 + 짧은 문구 또는 스켈레톤 후 리다이렉트.  
- Slack: 로고 + "Loading…" 이지만 브랜딩은 유지.

**개선 옵션**  
1. **로고 + "Loading…" (현재 스타일 유지, 로고만 추가)**  
   - 정합성: 헤더의 "Abstract"와 동일한 브랜딩.  
   - 구현: `layout` 또는 `page`에 로고/텍스트 로고 추가.  
2. **로고 + 한 줄 슬로건 + 로딩**  
   - 예: "Artist-centric community" + 스피너.  
   - 첫 인상 개선, 다만 `/` 체류 시간이 짧아 체감은 제한적.  
3. **랜딩 라이트**  
   - 비로그인 시 짧은 소개 + CTA(로그인/가입). 로그인 시 기존처럼 리다이렉트.  
   - 정합성: 아티스트·컬렉터 중심 플랫폼이라는 개발 철학과 잘 맞음. 다만 범위가 큼.

**순위**  
- 플랫폼 전체가 “미니멀·기능 우선”이므로 **1 → 2**. 랜딩은 별도 기획 시 **3**.

---

### 2.2 로그인 — Enter 키로 제출 불가

**현상**  
- 비밀번호 로그인 폼: `onSubmit`에서 `preventDefault()`만 하고, 버튼이 `type="button"` + `onClick`.  
- 이메일/비밀번호 입력 후 **Enter**를 눌러도 제출되지 않고, 반드시 "로그인" 버튼을 눌러야 함.

**벤치마크**  
- 대부분 서비스: 로그인 폼은 Enter로 제출.  
- 접근성·키보드 사용자 관례에 부합.

**개선 옵션**  
1. **폼 제출로 통일**  
   - `onSubmit={handlePasswordSignIn}`, 버튼 `type="submit"`.  
   - Enter로 로그인 가능.  
2. **유지**  
   - 실수 제출 방지 등 이유가 있으면 유지 가능하나, 로그인 폼에서는 보통 1이 관례.

**순위**  
- **1권장.** 변경 범위 작고, 플랫폼 전반의 “예상 가능한 폼 동작”과 정합성 높음.

---

### 2.3 로그인 — 이메일 입력이 두 군데

**현상**  
- 위: 비밀번호 로그인(이메일 + 비밀번호).  
- 아래: "이메일 링크로 로그인"(이메일 한 칸).  
- 같은 이메일을 두 번 입력할 수 있어, "위에서 쓴 이메일로 링크 보내기"가 자동으로 이어지지 않음.

**벤치마크**  
- GitHub, Vercel: 한 이메일 필드 → "비밀번호로 로그인" / "이메일 링크" 선택.  
- Linear: 로그인과 매직링크가 분리된 폼이지만, 일부 제품은 이메일 한 번만 입력.

**개선 옵션**  
1. **이메일 한 칸 공유**  
   - 상단에 이메일 한 칸, 그 아래 "비밀번호" 칸(있으면 입력) + [로그인].  
   - "비밀번호 없이 이메일 링크로 로그인" 링크 클릭 시 같은 이메일로 매직링크 발송.  
   - 정합성: 단일 진입점, 입력 감소.  
2. **현 구조 유지 + 안내 문구**  
   - "같은 이메일을 사용할 수 있습니다" 등.  
   - 개선 폭은 제한적.  
3. **매직링크를 보조로만**  
   - 비밀번호 로그인만 크게 두고, "이메일 링크로 로그인"은 작게.  
   - 현재도 비밀번호가 primary라 3은 이미 반영된 상태에 가깝고, 1이 더 나음.

**순위**  
- **1 → 2.** 1이 벤치마크·정합성 모두 좋음. 1이 작업량이 크면 2로 완화.

---

### 2.4 Auth 콜백 — 600ms 고정 대기

**현상**  
- "Signing you in..." 표시 후 `setTimeout(..., 600)` 후에 세션·프로필 확인 및 리다이렉트.  
- 이미 준비됐어도 600ms 기다려서 체감 지연이 생길 수 있음.

**벤치마크**  
- 대부분: 준비되는 대로 즉시 리다이렉트.  
- 최소 표시 시간을 두는 패턴(예: 300ms)은 있으나, 600ms는 다소 김.

**개선 옵션**  
1. **즉시 시도**  
   - 마운트 시 바로 `getSession` → `getMyProfile` → 조건별 `replace`.  
   - 대기 없이 진행.  
2. **최소 표시 시간만 짧게**  
   - 200~300ms 정도만 "Signing you in..." 유지 후 리다이렉트.  
   - 플래시 방지용.

**순위**  
- **1 권장.** 매직링크 UX 개선 철학(불필요한 대기 제거)과 맞음.  
- 깜빡임이 거슬리면 **2**.

## 3. 온보딩

### 3.1 온보딩 프로필 모드 — 라벨·버튼 영문 하드코딩

**현상**  
- "Username *", "Display name", "Main role", "Roles * (at least one)", "Select", "Continue", "Saving..." 등이 `t()` 없이 영문.  
- locale이 한국어여도 이 구간만 영문으로 나와 **일관성 깨짐**.

**벤치마크**  
- 다국어 서비스는 같은 화면 내에서 한 언어로 통일.  
- Abstract는 이미 i18n을 쓰고 있으므로, 온보딩도 키로 통일하는 것이 정합성 있음.

**개선 옵션**  
1. **전부 i18n 키로 교체**  
   - `onboarding.labelUsername`, `onboarding.labelDisplayName`, `onboarding.labelMainRole`, `onboarding.labelRoles`, `onboarding.select`, `onboarding.continue`, `onboarding.saving` 등 추가 후 `t()` 적용.  
2. **필수만 먼저**  
   - 제목·설명·CTA(Continue)만 i18n, 나머지는 단계적으로.

**순위**  
- **1 권장.** 플랫폼의 i18n 철학과 정합성, 그리고 신규 유저(한국어 사용자) 경험 개선에 직결.

---

### 3.2 온보딩 signup — 일부 영문 하드코딩

**현상**  
- "Create your account", "Email *", "Password *", "Confirm password *", "Username *", "Display name", "Main role", "Roles * (at least one)", "Select", "Create account", "Creating account...", "Check your email" 등 하드코딩.  
- placeholder "you@example.com", "username" 등도 i18n 없음.

**개선**  
- 3.1과 동일하게 해당 문자열을 i18n 키로 옮기고 `t()` 적용.  
- **순위**: 3.1과 통합해서 **전면 i18n(1)** 권장.

---

### 3.3 온보딩 — "이미 계정 있으면 로그인" 링크 부재

**현상**  
- 프로필 완성 모드(매직링크 직후)에서 "이미 계정이 있으신가요? 로그인" 같은 링크가 없음.  
- 잘못 들어온 사용자가 로그인으로 돌아가기 어렵음.

**벤치마크**  
- 대부분 로그인/가입 화면에 "계정이 있으신가요? / 없으신가요?" 교차 링크 제공.

**개선 옵션**  
1. **프로필 완성 폼 하단에**  
   - "이미 계정이 있으신가요? 로그인" 링크 추가.  
2. **헤더에 로고 클릭 → 로그인**  
   - 현재 온보딩 페이지에 헤더가 없을 수 있음.  
   - 있으면 로고 → `/login` 연결만 해도 됨.

**순위**  
- **1 권장.** 비로그인 상태로 온보딩에 왔을 가능성은 낮지만, 일관성·탈출 경로 확보 차원에서 추가 권장.

## 4. set-password

### 4.1 set-password — 전부 영문

**현상**  
- "Set your password", "Set a password so you can sign in…", "New password", "Confirm password", "At least 8 characters", "Same as above", "Setting password...", "Set password" 모두 영문.  
- 한국어 사용자에게는 어색하고, 온보딩에서 "다음 화면에서 비밀번호 설정"과 톤이 달라 보일 수 있음.

**개선**  
- 메시지·라벨·버튼을 i18n 키로 넣고 한/영 메시지 추가.  
- **순위**: 온보딩 i18n과 함께 **1순위**로 처리 권장.

---

### 4.2 set-password — "건너뛰기" 없음

**현상**  
- 매직링크만 쓰고 비밀번호를 두지 않으려는 사용자도 있는데, 현재는 비밀번호 설정 후에만 피드로 감.  
- (이미 구현된) `HAS_PASSWORD_KEY`가 없으면 `/set-password`로 보내는 정책과 맞지만, UX적으로 "나중에 설정" 선택지가 없음.

**벤치마크**  
- Slack, Notion 등: "Set password" 화면에서 "Skip for now" 제공.  
- 보안 정책상 필수라면 스킵 불가가 맞음.

**개선 옵션**  
1. **"나중에 설정" 링크**  
   - "Skip for now" 클릭 시 `HAS_PASSWORD_KEY`를 "skipped" 등으로 두고 `/feed`로.  
   - 이후 로그인 시 다시 set-password 유도 가능.  
2. **유지**  
   - 비밀번호 설정을 전제로 한 정책이면 유지.  
   - 다만 문구로 "다음 로그인부터는 이메일 링크 없이 로그인할 수 있어요" 등 이점을 명확히 하는 것은 권장.

**순위**  
- 정책이 "비밀번호 권장이되 필수는 아님"이면 **1**.  
- "반드시 설정"이면 **2 + 문구 보강**.

---

## 5. 공통·로딩·일관성

### 5.1 Loading / Signing you in — 영문 하드코딩

**현상**  
- `/` (홈), AuthGate, auth/callback, login Suspense, people, feed 등에서 "Loading...", "Signing you in..." 등이 영문.  
- locale이 한국어여도 로딩 구간만 영문.

**개선**  
- `common.loading`, `auth.signingIn` 등 키를 두고, 모든 로딩/인증 대기 문구를 `t()`로 통일.  
- **순위**: i18n 정합성을 위해 **1순위**로 처리 권장.

---

### 5.2 AuthGate 로딩

**현상**  
- 인증 필요 페이지에서 AuthGate가 "Loading..." 한 줄만 풀스크린으로 표시.  
- 어떤 서비스인지(로고) 없이 로딩만 보임.

**개선 옵션**  
1. **로고 + Loading**  
   - 상단에 Abstract 로고, 아래 "Loading..." (또는 `t("common.loading")`).  
2. **현재 유지**  
   - 체류 시간이 짧으면 최소한만 두어도 됨.

**순위**  
- **1**이 브랜딩·일관성 측면에서 유리.  
- 홈(/) 개선과 같이 적용하면 좋음.

---

## 6. 피드·네비게이션

### 6.1 Feed — 빈 상태(empty state)

**현상**  
- **팔로잉 탭** 빈 경우: 이미 `feed.followingEmptyTitle` + [People] CTA 버튼 있음.  
- **전체 탭** 빈 경우: `feed.noArtworks` 문구만 있고 **CTA(업로드·People) 없음**.  
- 신규 유저가 "전체"에서 빈 피드를 보면 다음 행동이 불명확할 수 있음.

**벤치마크**  
- Instagram, Twitter: 빈 피드 시 "팔로우할 사람 찾기", "첫 포스트 올리기" 등 CTA.  
- Abstract는 "업로드" / "사람 찾기"가 자연스러운 CTA.

**개선 옵션**  
1. **전체 탭 빈 상태에 CTA 추가**  
   - "아직 피드가 비어 있어요. 작품을 올리거나 사람을 찾아보세요." + [업로드] [People].  
2. **문구만 보강**  
   - "첫 작품을 올리거나 People에서 아티스트를 찾아보세요." 등으로 유도만.

**순위**  
- **1 권장.** 팔로잉 탭과 동일하게 CTA를 두면 정합성 있고, 신규 유저 이탈 방지·참여 유도에 맞음.

---

### 6.2 헤더 — "내 프로필" 진입점

**현상**  
- "내 프로필" 클릭 시 `profileUsername ? "/my" : "/onboarding"`.  
- 온보딩 완료 전에는 `/onboarding`으로 가는데, 이미 프로필이 있는데 username이 아직 안 나온 경우 등 엣지가 있을 수 있음.  
- 설정은 아바타 메뉴 안에 있어, "설정"을 찾는 사용자는 아바타 클릭이 필요함. (일반적인 패턴과 동일.)

**개선**  
- 현재 로직 유지해도 되고, "설정" 링크를 아바타 메뉴 상단에 두어 발견성을 높이는 정도만 해도 됨.  
- **순위**: 문제가 크게 보고되지 않으면 **낮은 우선순위**.

---

## 7. 업로드

### 7.1 업로드 — 의도(4가지) 부담

**현상**  
- 단일 업로드 시 "내 작품", "소장 작품", "갤러리 - 전시 & 보유", "큐레이션 작품" 4가지.  
- 신규 유저에게는 용어·차이가 쉽게 와닿지 않을 수 있음.

**벤치마크**  
- Art Basel, Artsy: 작품 등록 시 "작가 본인" / "갤러리 소유" 등 단순 분류.  
- Abstract는 프로비넌스·전시와 연결되므로 4가지가 필요할 수 있으나, **설명 툴팁이나 단계별 안내**로 부담 완화 가능.

**개선 옵션**  
1. **각 의도에 한 줄 설명**  
   - 예: "큐레이션 작품 — 전시·갤러리에서 큐레이션한 작가의 작품."  
2. **첫 업로드만 단계 안내**  
   - "이 작품은 누구의 작품인가요?" → 선택지 축약 또는 플로우 분리.  
3. **현재 유지**  
   - 이미 i18n 라벨이 있으므로, 1만 추가해도 개선.

**순위**  
- **1 권장.** 개발 철학(프로비넌스·역할 명확화)은 유지하면서 인지 부하만 줄이는 방향.

---

### 7.2 업로드 레이아웃 — 전시 만들기 탭

**현상**  
- "전시 만들기" 탭이 `/upload/exhibition`이 아니라 `/my/exhibitions/new?from=upload`로 이동.  
- 탭 이름이 "업로드"에 있지만 실제로는 전시 생성 페이지로 가서, "업로드"와 "전시"가 한 흐름으로 느껴질 수 있음.  
- 의도적으로 "업로드 진입점에서 전시도 만들 수 있게"한 설계라면 유지해도 됨.

**개선 옵션**  
1. **탭 라벨을 "전시 만들기"로 유지**  
   - 현재와 동일.  
2. **라벨을 "전시" 등으로 짧게**  
   - "업로드"와 구분만 명확히.  
3. **업로드에서 제거하고, 내 전시/헤더에서만 전시 만들기**  
   - 업로드 플로우를 "작품"에만 집중.

**순위**  
- 현재 설계(업로드 허브에서 전시 생성 유도)를 유지한다면 **1 또는 2**.  
- 업로드 단순화를 우선하면 **3**.

## 8. 설정

### 8.1 설정 — Test RPC 버튼 노출

**현상**  
- 설정 페이지 하단에 "Test RPC" 버튼이 개발/디버그용으로 보이는데, `NODE_ENV`로 숨기지 않고 항상 렌더됨.  
- 프로덕션에서 일반 사용자가 보면 혼란스럽고, 클릭 시 "RPC OK" 등만 보여도 의미를 모름.

**개선**  
- `process.env.NODE_ENV === "development"`일 때만 `TestRpcButton` 렌더.  
- **순위**: **1순위.** 프로덕션 품질·신뢰도와 직결.

---

### 8.2 설정 — BuildStamp

**현상**  
- 설정·헤더에 BuildStamp(빌드 시점 등)가 있음.  
- 개발/운영 확인용이라면 개발 환경에서만 보이게 하는 것이 일반적.

**개선**  
- 개발 시에만 표시하거나, 설정의 "고급" 섹션 안으로 옮기는 등.  
- **순위**: 8.1보다 낮아도 됨. 사용자에게 필수 정보는 아님.

---

## 9. 기타 페이지

### 9.1 초대(위임) 랜딩 — 로그인 후 복귀

**현상**  
- `/invites/delegation?token=…` 에서 "로그인하여 수락" 시 `/login?next=...` 로 가고, 로그인 후 `next`로 복귀.  
- 이미 구현된 패턴이고, 벤치마크와도 맞음.

**개선**  
- 특별한 개선 없이 유지.  
- 필요 시 "수락하면 전시 작품 추가 페이지로 이동합니다" 등 한 줄 안내 추가 정도.

---

### 9.2 공개 프로필(/u/[username]) — 404·비공개

**현상**  
- 존재하지 않는 username 또는 비공개 프로필 시 어떤 메시지를 보여주는지(친절한 404 vs 단순 에러) 확인하면 좋음.  
- 신규 유저가 다른 사람 프로필 링크를 눌렀을 때 경험이 중요.

**개선**  
- "해당 사용자를 찾을 수 없어요" / "비공개 프로필이에요" 등 명확한 문구 + 홈/피드로 돌아가기.  
- **순위**: 에러 메시지 정리 시 함께 처리.
