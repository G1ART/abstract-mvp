# Abstract MVP — UI/UX 감사 및 개선 제안 (2026-02)

현재 기능을 저해하지 않고, 복잡도를 높이지 않는 범위에서 정리했습니다.  
**확인·선택이 필요한 항목**은 §7에 모았고, **검색**은 §3.4와 §9에 요구사항·구현 방안을 구체적으로 반영했습니다(People: 사람 + 작품/테마 → 아티스트; 전시: 선정 작가 작품 리스트 + 리스트 내 검색; 추후: 메인 피드 자연어 검색).

**적용 이력:** §8 항목 1–6(뒤로가기 통일, i18n, nav.myProfile, 로딩 통일, 빈 상태 스타일 가이드) 적용 완료. 검색(§9)은 별도 구현 단계.

---

## 1. 잘 되어 있는 부분 (유지 권장)

- **네비게이션 구조**: Feed / People / Upload / My Profile이 명확하고, 아바타 드롭다운에 설정·위임·로그아웃이 잘 묶여 있음.
- **피드 탭**: All / Following + Latest / Popular — 역할이 분명함.
- **프로필 페르소나 탭**: 전시·CREATED·OWNS·All 등 역할별로 구분되고, /my에서 탭 순서 저장 가능.
- **위임 플로우**: 검색 초대 + 이메일 초대, 수락/거절, 해지/초대취소가 한 페이지에서 처리됨.
- **작품 상세 뒤로가기**: `getArtworkBack()`로 진입 경로(피드/내 프로필/사람 등) 복귀 — 일관적.
- **Acting-as 배너**: 위임 관리 시 “관리 중: {이름}” + “내 계정으로 전환”이 눈에 잘 띔.

## 2. 일관성 개선 (낮은 리스크)

### 2.1 뒤로가기 링크 문구 통일

**현재:**  
- 대부분: `← {t('common.backTo')} {대상}` (예: “← Back to My Profile”)  
- 일부만 “← My Profile” (backTo 없음): `/my/inquiries`, `/my/following`, `/my/followers`  
- 하드코딩/비 i18n: “← Back to artwork”, “← Single upload”, “Back to sign in”, “Back to login”, “← Back to Home”

**제안:**  
- 모든 “뒤로” 링크를 **`← {t('common.backTo')} {대상}`** 패턴으로 통일.  
- 대상은 i18n 키 사용 (예: `nav.myProfile`, `exhibition.myExhibitions`, `nav.feed`).  
- “Back to artwork” → `common.backTo` + artwork 상세용 키(예: `artwork.backToArtwork`) 추가.  
- “Single upload” → `upload.backToSingleUpload` 또는 기존 `upload.backToUpload`로 통일.  
- “Back to sign in” / “Back to login” / “Back to Home” → 각각 i18n 키 하나씩 정의해 사용.

**효과:** 다국어·톤앤매너 통일, 예측 가능한 UX. 기존 동작은 그대로 유지.

---

### 2.2 빈 상태(Empty state) 패턴 통일

**현재:**  
- 문구는 대부분 i18n (`priceInquiry.empty`, `my.claimsEmpty`, `exhibition.emptyList`, `people.noSearchResults` 등).  
- 스타일은 페이지마다 다름: 어떤 곳은 CTA 버튼 있음, 어떤 곳은 텍스트만.

**제안:**  
- **문구**: 유지.  
- **스타일**: “콘텐츠 없음”일 때 공통 패턴 1가지만 정함.  
  - 예: `rounded-lg border border-zinc-200 bg-zinc-50/50 p-6 text-center` + 메시지 + (해당되면) CTA 버튼 1개.  
- 새 페이지에서만 이 패턴 적용하고, 기존 페이지는 리팩터 시점에 점진적으로 맞춤.  
- **선택지:** 공통 `EmptyState` 컴포넌트를 만들지, 아니면 클래스/스타일만 가이드로 문서화할지 결정 필요.

**스타일 가이드 (문서화·적용 시 참고):**  
콘텐츠 없음일 때 공통 스타일: `rounded-lg border border-zinc-200 bg-zinc-50/50 p-6 text-center` + 메시지(i18n) + (해당 시) CTA 버튼 1개. 새 페이지 또는 리팩터 시 이 패턴을 사용한다.

---

### 2.3 로딩 문구

**현재:** `feed.loading`, `people.loading`, `common.loading` 혼용.

**제안:**  
- 페이지 로딩은 모두 `common.loading` 사용.  
- 피드/사람은 이미 해당 키를 쓰고 있으므로, 값만 동일하게 맞추거나 키를 `common.loading`으로 바꿔도 됨.  
- 버튼 로딩(“제출 중”)은 기존처럼 “...” 또는 `common.loading` 유지.

## 3. 발견성·명확성 개선 (중간 리스크 없음)

### 3.1 “내 프로필(/my)” 진입점

**현재:**  
- 헤더에서 “My Profile” 클릭 시 `/my` 또는 (유저네임 없으면) `/onboarding`.  
- 피드/다른 곳에서 “내 대시보드”로 가는 추가 진입점은 없음.

**상태:** 이미 명확함. 변경 제안 없음.

---

### 3.2 /my 대시보드 KPI 블록

**현재:**  
- 팔로잉 / 팔로워 / 게시물 / 가격 문의 / 클레임 요청 5개.  
- 팔로잉·팔로워·가격 문의·클레임 요청은 클릭 시 각각 `/my/following`, `/my/followers`, `/my/inquiries`, `/my/claims`로 이동.  
- “게시물”만 링크 없음(숫자만 표시).

**제안:**  
- “게시물”도 클릭 시 스크롤을 아래로 내려 **같은 페이지의 작품 그리드(또는 All 탭)** 로 포커스하거나, 또는 그대로 두고 “지표만 표시”로 명확히 할 수 있음.  
- **선택지:** (A) 게시물 클릭 시 페이지 내 앵커로 이동 (id 부여 후 scroll-into-view) / (B) 현재처럼 비링크 유지.  
- 툴팁/보조 문구(“클릭하면 작품 목록으로”)를 넣을지는 선택에 따라 결정.

### 3.3 업로드 영역 진입 경로

**현재:**  
- 헤더 “Upload” → `/upload` (개별 업로드).  
- /my에서 “새 작품 업로드” → `/upload`, “전시 만들기 및 관리” → `/my/exhibitions`.  
- /upload 레이아웃에 탭: 개별 / 벌크 / 전시 만들기.

**제안:**  
- 현재 구조 유지.  
- “전시 만들기”가 `/upload` 탭과 `/my/exhibitions` 양쪽에 있는데, 역할이 다름(업로드 플로우 중 전시 생성 vs 전시 목록·관리).  
- **선택지:** (A) 그대로 두기 — “업로드 컨텍스트에서 전시 만들기”와 “전시 관리 홈에서 만들기” 둘 다 유지. (B) “전시 만들기”는 `/my/exhibitions` 쪽으로만 두고, /upload 탭에서는 “전시 만들기” 탭 제거 후 “전시에 작품 추가” 링크만 두기.  
- (B) 시 탭 수 감소으로 단순해지지만, “업로드하면서 바로 전시 생성” 경로가 사라짐. 필요 여부 확인 필요.

### 3.4 검색 범위 및 전략

**요구사항 정리:**

- **People 탭 검색:**  
  - 유저가 사람 이름이 아니라 **작품·테마**(예: 달항아리)로 생각하는 경우가 많음.  
  - “이 테마로 작업하는 작가를 한눈에 보고 싶다”는 니즈를 만족하려면, **작품/테마 기반 검색 → 결과는 사람(아티스트)** 이어야 함.  
- **전시 작품 추가 시 검색:**  
  - 전시 게시물 생성 후 **온보딩된 작가를 선정**했을 때, **해당 작가의 작품 썸네일 리스트**를 보여주고, **그 리스트 안에서 검색(필터)** 이 가능하면 당장은 충분함.  
- **추후(콜렉터·큐레이터 증가 시):**  
  - **메인 피드에서 작가·작품을 자연어로 바로 검색**하는 기능은 반드시 필요. 별도 로드맵으로 문서화.

**검색 범위 정의:**

| 맥락 | 검색 대상(입력) | 결과 | 비고 |
|------|------------------|------|------|
| **People 페이지** | (1) 이름·@유저네임 (2) 작품 제목·테마·키워드·매체 | **사람(프로필)** 목록 | (2)로 “달항아리” 검색 시 해당 작품을 가진 아티스트 목록 반환 |
| **전시 작품 추가** (`/my/exhibitions/[id]/add`) | 선정한 **온보딩 작가의 작품** 리스트 내에서의 텍스트 검색 | 해당 작가의 작품 썸네일(필터된 목록) | 당장은 “선정 작가의 작품 + 리스트 내 검색”만 구현하면 충분 |
| **추후: 메인 피드** | 자연어(작가명·작품명·테마 등) | 작가·작품 통합 검색 | 콜렉터/큐레이터 확대 후 구현; 본 문서에서는 요구사항만 명시 |

**제안:**  
- **People:** 기존 `search_people`(이름/유저네임) 유지 + **작품/테마 기반 아티스트 검색** 추가(신규 RPC 또는 기존 확장).  
- **전시 작품 추가:** 선정 작가가 온보딩 작가일 때 **해당 작가의 작품만** 불러와 썸네일 리스트로 표시하고, **리스트 내 검색(제목·매체·스토리·키워드)** 지원.  
- **메인 피드 자연어 검색:** 구현하지 않고, “추후 필수” 요구사항으로만 §9에 정리.

### 3.5 사람 추천 노출

**현재:**  
- **피드:** 추천 프로필 + 작품 미리보기 블록이 일정 간격으로 삽입.  
- **People:** 검색 없을 때 추천 레인(팔로우 그래프/좋아요 기반/확장).

- “추천이 없습니다”일 때 People에서 빈 상태 메시지 + (선택) “피드에서 더 보기” 등 보조 CTA는 나중에 고려.

## 5. 톤·표현 정리

### 5.1 “nav.me” vs “My Profile”

**현재:**  
- 위임 페이지 등 일부는 `nav.me`(“Me” / “내 정보”), 다른 곳은 “My Profile”(`nav.myProfile`).

**제안:**  
- “뒤로가기” 대상으로는 **한 가지로 통일**.  
- **선택지:** (A) 내 대시보드/내 프로필 관련 뒤로가기는 모두 `nav.myProfile` (“My Profile” / “내 프로필”). (B) 모두 `nav.me` (“Me” / “내 정보”).  
- (A)가 “프로필”이라는 맥락을 더 드러내서, 뒤로가기에는 (A) 권장.

---

### 5.2 에러·토스트

**현재:**  
- 에러: 인라인 빨간 텍스트.  
- 성공: 페이지별 로컬 상태 + 고정 위치 또는 인라인 메시지.  
- 공통 토스트/스낵바 컴포넌트 없음.

**제안:**  
- 단기: 기존 방식 유지.  
- 중기: 공통 토스트(예: 우하단, 2–3초) 도입 시 “저장됨”/“초대 보냈습니다” 등 성공 메시지를 한 패턴으로 모으면 좋음.  
- **지금 단계:** 새 기능 추가 시 가능하면 비슷한 위치(예: 페이지 상단/카드 하단)와 스타일로 메시지 표시만 맞추기.

## 6. 복잡도·시각적 부담

- **탭 수:**  
  - 피드: All/Following + Latest/Popular → 유지.  
  - /my 프로필: 페르소나 탭이 많을 수 있으나, 역할별로 필요하므로 유지.  
  - 업로드: 개별/벌크/전시 3개 → 위 3.3에서 (B) 선택 시 2개로 줄일 수 있음.  
- **버튼 수:**  
  - /my 상단 CTA(업로드, 전시 만들기 및 관리, 아티스트 찾기, Reorder 등)는 역할이 달라서 그대로 두는 것을 권장.  
- **헤더:**  
  - 로그인 시 Feed / People / Upload / My Profile + 아바타 메뉴로 적당함. 항목 추가는 자제.

## 7. 확인·선택이 필요한 항목 (정리)

아래는 제품/정책 결정이 있으면 좋은 부분입니다. 선택하시면 그에 맞춰 구현 범위를 정할 수 있습니다.

1. **뒤로가기 통일**  
   - “← Back to artwork”, “← Single upload”, “Back to sign in”, “Back to login”, “Back to Home”을 모두 `common.backTo` + i18n 키로 바꿀지 여부.  
   - **권장:** 예. (작업량 적음, 일관성 확보.)

2. **빈 상태(Empty state)**  
   - 공통 `EmptyState` 컴포넌트를 만들지, 아니면 스타일 가이드만 문서로 둘지.  
   - **권장:** 먼저 스타일 가이드만. 컴포넌트는 빈 상태가 더 늘어날 때 도입.

3. **/my “게시물” KPI**  
   - (A) 클릭 시 같은 페이지의 작품 그리드로 스크롤 이동  
   - (B) 현재처럼 링크 없이 숫자만 표시  
   - **권장:** (B) 유지. “게시물”은 요약 수치로 두고, 실제 목록은 아래 탭으로 이미 노출됨.

4. **업로드 “전시 만들기” 탭**  
   - (A) 현재처럼 /upload에 “전시 만들기” 탭 유지  
   - (B) /upload에서는 “전시 만들기” 탭 제거, 전시 생성은 /my/exhibitions 쪽으로만  
   - **권장:** (A). “업로드하면서 전시도 만들기” 시나리오를 지금도 쓰고 있다면 유지.

5. **“Me” vs “My Profile”**  
   - 뒤로가기·페이지 제목 등에서 `nav.me` vs `nav.myProfile` 중 어떤 라벨을 기본으로 쓸지.  
   - **권장:** 뒤로가기/내 영역 제목은 `nav.myProfile`(“내 프로필”)로 통일.

6. **검색**  
   - 검색 범위·구현 방향은 §3.4와 §9에 구체적으로 정리됨. (People: 사람 + 작품/테마 → 아티스트; 전시: 선정 작가 작품 리스트 내 검색; 추후: 메인 피드 자연어 검색.)

## 9. 검색 기능 구현 방안 (구체)

권장 방향(§3.4)을 구현하기 위한 구체적 작업 단위입니다. 기존 동작을 깨지 않도록 단계별로 적용합니다.

### 9.1 People 페이지: 작품/테마로 아티스트 검색

**목표:**  
People 검색창에 “달항아리”, “도자기”, “abstract painting” 등 **작품 제목·테마·키워드·매체**를 입력하면, 해당 내용이 들어간 **작품을 보유한 아티스트(프로필)** 목록이 나온다.  
(이름/유저네임 검색은 기존처럼 유지.)

**백엔드**

- **신규 RPC** (예: `search_artists_by_artwork`)  
  - 인자: `p_q text`, `p_roles text[]`(선택), `p_limit int`, `p_cursor text`(선택).  
  - 동작:  
    - `artworks`(및 필요한 메타: title, medium, story, keywords 등) + `artist_id`로 매칭.  
    - `visibility = 'public'` 등 노출 조건 적용.  
    - 제목·매체·스토리·키워드에 `ilike` 또는 full-text로 `p_q` 매칭.  
    - 매칭된 작품의 `artist_id`로 distinct한 프로필만 반환.  
  - 반환: 기존 `search_people`과 같은 프로필 스키마(jsonb)로 통일하면 People 클라이언트 재사용이 쉬움.  
- 또는 **기존 `search_people` 확장:**  
  - `p_mode text`(예: 'name' | 'artwork') 추가.  
  - `p_mode = 'artwork'`일 때는 위 로직(작품 기반 → artist_id distinct → 프로필) 수행.  
  - 기존 이름 검색은 `p_mode = 'name'` 또는 기본값으로 유지.

**프론트(People)**

- 검색 입력 1개 유지.  
- **전략 (택 1):**  
  - **(A)** 입력 시 `search_people(q)`와 `search_artists_by_artwork(q)`(또는 `search_people(q, ..., mode: 'artwork')`) **둘 다** 호출 후, 결과를 **합치고 중복 제거(profile id 기준)** 하여 한 목록으로 표시.  
  - **(B)** 탭 또는 토글 “이름으로 / 작품·테마로” 선택 시, 선택에 따라 한 API만 호출.  
- **(A)**가 “이름이든 테마든 한 번에 검색” 느낌을 주므로 UX상 권장.  
- 역할 필터는 그대로 적용(반환된 프로필을 클라이언트에서 필터하거나, RPC에 `p_roles` 전달).

**i18n**

- placeholder 예: “이름, @유저네임 또는 작품·테마로 검색” / “Search by name, @username, or artwork/theme”.  
- 빈 결과 메시지: 기존 `people.noSearchResults` 유지 또는 “이름이나 작품·테마를 바꿔 보세요” 등으로 확장.

### 9.2 전시 작품 추가: 선정 작가의 작품 리스트 + 리스트 내 검색

**목표:**  
전시 게시물 생성 후 **참여 작가(온보딩)** 를 선정했을 때, **그 작가(들)의 작품만** 썸네일 리스트로 보여주고, **리스트 내에서 제목·매체·스토리·키워드로 검색(필터)** 할 수 있게 한다.

**영향 범위 (새 작품 업로드와의 관계)**  
- 9.2에서 바꾸는 것은 **“기존 작품 풀”의 정의**뿐이다(참여 작가가 있으면 → 선정 작가들의 공개 작품만 풀로 사용).  
- **새 작품 업로드 후 전시에 추가**하는 흐름(`/upload?addToExhibition=...` 링크로 업로드 → 완료 후 전시에 추가)은 **전혀 건드리지 않는다.**  
- “기존 작품에서 고르기”와 “새로 업로드해서 넣기”는 계속 병행 가능하다.

**현재 상태**

- `/my/exhibitions/[id]/add`:  
  - 1단계 참여 작가 선택, 2단계 작품 선택.  
  - 작품 풀은 현재 **큐레이터 본인 작품 + 본인이 리스팅한 작품** (`listMyArtworks` + `listPublicArtworksListedByProfileId`).  
  - `workQuery`로 제목·매체·스토리·키워드 필터링 가능.  
  - “작가별 필터”로 참여 작가 중 한 명만 골라서 보는 것은 가능.

**필요한 변경**

- **데이터 소스 분기:**  
  - **참여 작가가 1명 이상 있을 때:**  
    - “이 전시에 넣을 수 있는 작품”을 **선정된 온보딩 작가들의 공개 작품**으로 제한.  
    - 즉, `participants`의 각 `id`에 대해 **해당 프로필이 artist인 공개 작품**을 가져오는 API 사용(예: 기존 `listPublicArtworksForProfile(profileId)` 또는 전시용 `list_works_by_artists(artist_ids[])` 같은 RPC).  
  - 참여 작가가 없거나 “내 작품 + 리스팅”만 쓸 때는 **기존처럼** `listMyArtworks` + `listPublicArtworksListedByProfileId` 유지.  
- **리스트 내 검색:**  
  - 위에서 가져온 “선정 작가 작품” 리스트에 대해, **클라이언트 또는 서버**에서 `workQuery`(제목·매체·스토리·키워드) 필터 적용.  
  - 이미 있는 `workQuery` 입력란과 `filteredArtworks` 로직을 그대로 활용하면 됨.  
- **외부 작가(미온보딩):**  
  - 이메일 초대만 있고 “그 작가의 작품 리스트”는 없으므로, 해당 참여자에 대해서는 기존처럼 “내가 리스팅한 그 사람 작품” 또는 “직접 업로드 후 추가” 플로우 유지.

**구현 순서 제안**

1. 백엔드: “여러 artist_id에 대한 공개 작품 목록” 반환 RPC 또는 기존 `listPublicArtworksForProfile`를 여러 번 호출해 병합.  
2. `/my/exhibitions/[id]/add` 2단계에서, `participants.length > 0`이면 위 “선정 작가 작품”을 풀로 사용하고, 기존 `workQuery`로 필터.  
3. 참여 작가 0명이면 기존처럼 “내 작품 + 리스팅” 풀 유지.  
4. UI: “선정한 작가의 작품만 보기”임을 한 줄 안내 문구로 표시(선택).

### 9.3 추후: 메인 피드 자연어 검색 (작가·작품)

**요구사항만 명시 (구현 시점: 콜렉터·큐레이터 확대 후)**

- **메인 피드(또는 전역)** 에서 **자연어**로 “작가” 또는 “작품” 검색.  
- 예: “김홍도”, “달항아리”, “abstract painting” 입력 시 작가 목록·작품 목록(또는 통합 결과)을 한눈에.  
- 구현 형태는 미정:  
  - 전용 검색 페이지,  
  - 피드 상단 검색창 + 결과 오버레이/페이지,  
  - 또는 헤더 검색 아이콘 → 모달/드롭다운.  
- 이 단계에서는 **로드맵 항목**으로만 두고, §9.1·9.2 완료 후 필요 시 별도 스펙으로 상세화.
