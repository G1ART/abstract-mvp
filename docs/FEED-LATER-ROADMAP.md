# 피드: 나중에 구현할 것 & 임계치 가이드

이 문서는 **지금 구현해 두면 초기 운영에 해가 되는** 항목은 하지 않고, **유저·데이터·행위가 일정 수준 이상**일 때 도입할 항목과, 그 시점을 결정하기 위한 **임계치 벤치마크**를 정리한 것이다.  
피드 퍼포먼스·개인화 전략 전체는 [FEED-PERFORMANCE-RECOMMENDATIONS.md](./FEED-PERFORMANCE-RECOMMENDATIONS.md) 참고.

---

## 1. 지금 이미 적용한 것 (초기 운영에 무리 없음)

- **userId 없을 때 fetch 스킵**: FeedContent는 userId가 확정된 뒤 한 번만 fetch. 중복 풀 로드 제거.
- **Discovery 병렬화**: 추천 프로필 4명분 `listPublicArtworksForProfile`을 순차 4회 → `Promise.all` 1회로 변경.
- **스켈레톤 UI**: 로딩 중 "Loading..." 대신 그리드 형태 스켈레톤 카드 12개 표시.

→ 데이터/노출 로직은 그대로이고, 속도·체감만 개선되므로 초기 운영에 해가 되지 않음.

---

## 2. 나중에 구현할 것 (임계치 도달 시)

아래 항목은 **유저·데이터·행위가 일정 수준 이상**일 때 도입하는 것을 권장한다. 너무 일찍 넣으면 복잡도·버그 리스크만 커지고 체감 이득이 작다.

| 항목 | 설명 | 참고 |
|------|------|------|
| **유저별 메인 피드 랭킹** | 메인 리스트(전체 탭)를 전역 최신/인기 대신 유저별로 정렬 (온보딩·위치·좋아요/팔로우/가격문의 등 행위 기반). 백엔드 RPC `get_feed_first_page(user_id, ...)` 등으로 첫 페이지를 유저별 구성. | 섹션 3 임계치 참고 |
| **첫 페이지만 로드 + 더보기** | 첫 로딩 시 작품 18~24·전시 6~12 수준만 요청하고, 나머지는 "더 보기"/무한스크롤. LCP 고정 + 백엔드가 커서/랭킹으로 슬라이스 제공. | 콘텐츠가 많아져서 50+30 한 번에 가져오기가 무거워질 때 |
| **단계적 로딩(Phase1/2 분리)** | Phase1: 작품+전시만 먼저 표시 → Phase2: 좋아요·discovery 병렬로 이어서 합치기. (선택) 현재는 한 번에 다 받아도 무리가 없으면 유지 가능. | 첫 페이지만 로드와 함께 도입하면 자연스러움 |

---

## 3. 임계치 벤치마크 (언제 "나중에 구현"을 실행할지)

다른 플랫폼·추천 시스템 사례를 참고한 **권장 임계치**다. 절대값이라기보다 "이 정도 되면 개인화/첫 페이지 제한을 검토해도 된다"는 트리거로 쓰면 된다.

### 3.1 참고한 근거

- **Amazon Personalize** 등 상품 추천:
  - **기본**: 최소 **약 1,000건의 아이템 상호작용** + **25명 이상 유저**가 각각 2회 이상 상호작용.
  - **품질 추천**: **약 50,000건 상호작용** + **1,000명 이상 유저**가 각각 2회 이상.
  (출처: AWS Personalize 인터랙션 데이터 가이드)
- **협업 필터링/콜드스타트**:
  - "최소 N명/N개"에 대한 통일된 업계 기준은 없고, 데이터 특성·알고리즘에 따라 다름.
  - 다만 **상호작용이 너무 적으면** 개인화 품질이 나오지 않고, **너무 일찍** 복잡한 랭킹을 넣으면 오히려 노이즈·버그 가능성이 커진다는 연구가 많음.
- **인스타/핀터레스트 등**:
  - "알고리즘 전환 시점"의 **명시적 MAU/콘텐츠 수 임계치**는 공개 자료에서 잘 안 나옴.
  - 다만 소규모 커뮤니티(예: 수백만 MAU 수준)에서도 크로노 피드가 engagement에 유리하다는 비교가 있고, "데이터가 풍부해진 뒤" 알고리즘 피드로 전환하는 사례가 일반적.

### 3.2 Abstract 피드에 적용할 권장 임계치

| 지표 | 검토 시작 (기획·설계) | 본격 도입 권장 |
|------|------------------------|-----------------|
| **가입 유저 수** | 약 **500~1,000명** | 약 **2,000~5,000명** |
| **공개 작품 수** | 약 **500~1,000점** | 약 **2,000~5,000점** |
| **공개 전시 수** | 약 **50~100개** | 약 **200~500개** |
| **유의미한 행위 수** (좋아요+팔로우+가격문의 등 합산) | 약 **2,000~5,000건** | 약 **10,000~25,000건** |
| **유저당 행위** (평균) | 유저 25% 이상이 좋아요/팔로우/문의 중 **2회 이상** | 유저 50% 이상이 **5회 이상** |

- **"검토 시작"**: 위 수준에 도달하면 유저별 메인 피드·첫 페이지만 로드·단계적 로딩 등을 **기획·백엔드 설계** 단계에서 다루기 좋은 시점.
- **"본격 도입 권장"**: 두 번째 열 수준이 되면, 실제로 **유저별 랭킹/첫 페이지 API**를 넣었을 때 체감 품질·성능 이득이 나오기 쉬운 구간.

### 3.3 운영 시 확인 방법

- 대시보드 또는 Supabase에서 주기적으로:
  - `profiles` 수(또는 최근 로그인 기준 활성 유저),
  - 공개 `artworks` / 전시(프로젝트) 수,
  - `artwork_likes`, `follows`, 가격 문의 등 **행위 테이블 row 수**
  를 보고, 위 표와 비교해 "나중에 구현" 항목을 언제 실행할지 결정하면 된다.
