# 온보딩·난수 아이디 UX 플로우

초기 가입 시 유저가 겪는 단계와, 이미 온보딩을 마쳤지만 난수 아이디(`user_xxxxxxxx`)가 배정된 유저가 보게 되는 화면·행동을 UX 플로우 형태로 정리한 문서.

---

## 1. 초기 온보딩 — 매직링크 경로

**대상:** 로그인 페이지에서 이메일만 입력하고 “매직 링크 보내기”로 처음 가입하는 유저.

| 단계 | 화면 / 유저 행동 | 시스템 동작 | 비고 |
|------|------------------|-------------|------|
| 1 | **로그인 페이지** (`/login`) | — | |
| | 유저: 이메일 입력 후 **「매직 링크 보내기」** 클릭 | `sendMagicLink(email)` 호출 | 비밀번호 없음 |
| 2 | **「이메일을 확인하세요」** 안내 표시 | 메일 발송 완료 | 링크 클릭 전까지 대기 |
| 3 | 유저: 메일함에서 매직 링크 클릭 | 브라우저가 `/auth/callback`(또는 `?next=...` 있으면 해당 URL)으로 이동 | |
| 4 | **Auth 콜백** 화면: “Signing you in...” | `getSession()` → `getMyProfile()` | |
| | | 프로필 **없음** (온보딩 경로에서는 `ensure_my_profile` 미호출로 프로필 행이 아직 없음) | |
| 5 | | `router.replace("/onboarding")` | |
| 6 | **온보딩 페이지** (`/onboarding`) | | |
| | 제목: **「프로필 완성」** | | |
| | 안내: **「유저 아이디와 공개 이름을 입력하세요. 추가 이메일 링크는 발송되지 않습니다.»** | | 매직링크 재발송 없음 명시 |
| | 폼 필드: | | |
| | · **유저 아이디 *** (3–20자, 소문자·숫자·언더스코어) | | |
| | · **표시 이름** (공개 이름) | | |
| | · **주요 역할** (Main role) | | |
| | · **역할 *** (최소 1개: artist, collector, curator, gallerist) | | |
| | 하단 안내: **「다음 로그인을 더 쉽게」** — “계속하기 후 다음 화면에서 비밀번호를 설정하면 이메일 링크 없이 로그인할 수 있습니다.” | | |
| 7 | 유저: 필수 항목 입력 후 **「Continue」** 클릭 | `saveProfileUnified` → `upsert_my_profile` (선택한 유저아이디·이름·역할로 프로필 생성) | **추가 이메일 발송 없음** |
| 8 | | 비밀번호 미설정(`HAS_PASSWORD_KEY` 없음)이면 → **`/set-password`** 로 이동 | 매직링크 유저는 보통 여기로 감 |
| 9 | **비밀번호 설정 페이지** (`/set-password`) | | |
| | 제목: “Set your password” | | |
| | 안내: 이메일·비밀번호로 로그인하려면 비밀번호를 설정하세요 (이메일 링크 불필요). | | |
| | 필드: 새 비밀번호, 비밀번호 확인 | | |
| 10 | 유저: 비밀번호 입력 후 제출 | `updateUser({ password })`, `HAS_PASSWORD_KEY` 저장 | **이메일 발송 없음** |
| 11 | | `router.replace("/feed?tab=all&sort=latest")` | **온보딩 완료** |

**요약:**  
매직링크 클릭 → 콜백 → **온보딩 1회** (유저아이디·공개 이름·역할 입력) → **비밀번호 설정 1회** → 피드.  
이 과정에서 **추가 매직링크나 인증 메일은 발송되지 않음.**

---

## 2. 초기 온보딩 — 이메일·비밀번호 가입 (온보딩 페이지에서)

**대상:** 로그인 페이지에서 “계정 만들기”로 넘어가거나, 비로그인 상태로 `/onboarding`에 직접 들어와 이메일·비밀번호로 가입하는 유저.

| 단계 | 화면 / 유저 행동 | 시스템 동작 | 비고 |
|------|------------------|-------------|------|
| 1 | **온보딩 페이지** (`/onboarding`) — 비로그인 | `getSession()` → 세션 없음 → `mode = "signup"` | |
| | 제목: “Create your account” | | |
| | 안내: “Use email and password to sign up without waiting for an email link.” | | |
| | 폼: 이메일 *, 비밀번호 *, 비밀번호 확인 *, **유저 아이디 ***, 표시 이름, 주요 역할, **역할 *** (최소 1개) | | |
| 2 | 유저: 모든 필드 입력 후 **「Create account」** 클릭 | `signUpWithPassword(email, password, { username, display_name, main_role, roles })` | |
| 3a | **이메일 확인이 꺼져 있는 경우** (즉시 세션 부여) | | |
| | | `saveProfileUnified`로 프로필 저장, `HAS_PASSWORD_KEY` 저장 | |
| | | `router.replace("/feed?tab=all&sort=latest")` | **온보딩 완료** (한 번에 피드) |
| 3b | **이메일 확인이 켜져 있는 경우** | | |
| | **「이메일을 확인하세요」** 화면 | “We sent a confirmation link to {email}. Click it to activate your account, then sign in.” | Supabase가 보내는 **회원가입 확인 메일** (매직링크 아님) |
| | 유저: 확인 링크 클릭 | Auth 트리거 등으로 프로필이 먼저 생성될 수 있음 → 이때는 `user_xxxxxxxx` 형태일 수 있음 | 기존 이슈: 확인 후 온보딩 폼 재노출 안 됨 |
| | | 이후 로그인 시 콜백 → 프로필 있음 → 비밀번호 없으면 `/set-password`, 있으면 `/feed` | |

**요약:**  
이메일·비밀번호 가입은 **한 페이지에서 이메일·비밀번호·유저아이디·표시이름·역할**을 한 번에 입력.  
확인 메일 사용 시에는 확인 링크 클릭 후 프로필이 이미 있어서 “프로필 완성” 폼이 다시 뜨지 않을 수 있음 (기존 감사 문서 참고).

## 3. 초기 온보딩 — 진입 경로별 정리

| 진입 경로 | 1단계 | 2단계 | 3단계 |
|-----------|--------|--------|--------|
| **매직링크** | 로그인 페이지에서 이메일 입력 → 링크 클릭 | 콜백 → **온보딩(프로필 완성)** — 유저아이디·이름·역할 | 비밀번호 없음 → **set-password** → 피드 |
| **이메일·비밀번호 (확인 없음)** | 온보딩 signup 폼에서 한 번에 입력 | Create account → **즉시 피드** | — |
| **이메일·비밀번호 (확인 있음)** | 온보딩 signup 폼 제출 → 확인 메일 | 확인 링크 클릭 → 콜백 → 피드 또는 set-password | “프로필 완성” 폼은 대개 스킵됨 (프로필 이미 생성) |

---

## 4. 이미 온보딩한 유저 — 난수 아이디(`user_xxxxxxxx`)가 배정된 경우

**대상:** 과거에 매직링크 또는 이메일 확인 등으로 가입했고, 프로필이 **자동 생성된 유저아이디** (`user_` + 8자 16진수)를 가진 유저.  
댓글·태깅·멘션 등에서 유저아이디가 필요하므로, **설정에서 유저아이디를 바꾸도록 권장**하는 플로우.

| 단계 | 화면 / 유저 행동 | 시스템 동작 | 비고 |
|------|------------------|-------------|------|
| 1 | 유저가 **로그인된 상태**로 서비스 이용 (피드, 설정, 업로드, 전시 등 **어떤 페이지든** 접속) | 레이아웃에서 `RandomIdBanner` 렌더 | |
| | | `getMyProfile()` → `username`이 `user_[a-f0-9]{8}` 패턴인지 확인 | |
| 2 | **배너가 보이는 조건** | `username`이 `user_xxxxxxxx` 형식이고, 같은 기기에서 **배너를 닫은 적이 없을 때** | `localStorage` 키 `ab_random_id_banner_dismissed` 없음 |
| 3 | **배너 UI** (헤더 바로 아래, 노란색 배경) | | |
| | · 문구(한): **「임시 아이디를 사용 중입니다»** | | |
| | · 문구(한): **「댓글·태깅을 위해 설정에서 유저 아이디를 설정하세요.»** | | |
| | · 버튼: **「설정으로 이동»** (`/settings` 링크) | | |
| | · **×** 닫기 버튼 | | |
| 4a | 유저: **「설정으로 이동」** 클릭 | `/settings`로 이동 | |
| | **설정 페이지** | 프로필 영역에 **유저 아이디(Username)**·표시 이름 등 편집 필드 있음 | |
| | 유저: 유저 아이디를 원하는 값으로 수정 후 저장 | `saveProfileUnified` → `upsert_my_profile` | |
| | | 다음 페이지 로드 시 `username`이 더 이상 `user_xxxxxxxx`가 아니므로 **배너 조건 불일치** → **배너 미표시** | |
| 4b | 유저: **×** 클릭 | `localStorage.setItem("ab_random_id_banner_dismissed", "1")` | |
| | | 배너가 사라짐. **같은 브라우저(기기)에서는 이후에도 배너가 다시 뜨지 않음** | 나중에 설정에서 바꿀 수 있음 |
| 5 | 배너를 닫지 않고 **설정에서 유저 아이디만 변경**한 경우 | 저장 후 다음 로드부터 배너 미표시 (패턴 불일치) | 닫기 여부와 무관 |

**요약:**  
- **보게 되는 것:** 헤더 아래 “임시 아이디를 사용 중입니다” + “설정에서 유저 아이디를 설정하세요” + [설정으로 이동] + [×].  
- **하게 되는 일:** (1) **설정으로 이동** → 설정에서 유저 아이디 수정 후 저장 → 이후 배너 안 뜸. (2) **×**로 배너만 닫기 → 같은 기기에서는 배너 재노출 안 함.

---

## 5. 플로우 다이어그램 (요약)

```
[초기 가입 - 매직링크]
  로그인(이메일) → 매직링크 클릭 → /auth/callback
    → 프로필 없음 → /onboarding
      → "프로필 완성" (유저아이디·이름·역할) → Continue
        → 비밀번호 없음 → /set-password → 비밀번호 설정 → /feed

[초기 가입 - 이메일·비밀번호]
  /onboarding (signup) → 이메일·비밀번호·유저아이디·이름·역할 → Create account
    → (확인 없음) 즉시 /feed
    → (확인 있음) 이메일 확인 → 링크 클릭 → 콜백 → /feed 또는 /set-password

[기존 유저 - 난수 아이디]
  로그인 상태로 아무 페이지 접속
    → username === user_xxxxxxxx 이고 배너 미닫음
      → 헤더 아래 배너: "임시 아이디 사용 중" + "설정에서 유저 아이디 설정" + [설정으로 이동] [×]
        → [설정으로 이동] → /settings → 유저 아이디 수정·저장 → 이후 배너 안 뜸
        → [×] → 배너 숨김 (같은 기기에서 재노출 안 함)
```

---

이 문서는 현재 구현(`ProfileBootstrap` 온보딩 스킵, `RandomIdBanner`, 온보딩 문구·리다이렉트)을 기준으로 작성되었으며, 콜백·로그인·설정 페이지의 실제 코드와 함께 보면 된다.
