# i18n·언어 통일 개선 방안

## 현재 이슈

1. **KO 선택 시에도 영문 노출**: 일부 메뉴·버튼·플레이스홀더가 여전히 영문 하드코딩되어 있음 (예: upload/settings/onboarding/artwork 편집 등).
2. **브라우저 자동 번역 충돌**: 사용자가 브라우저 번역을 켜면 앱이 제공하는 문자열까지 번역되어 "Feed" → "먹이", "Abstract" → "초록" 등 잘못된 결과가 나옴.
3. **en/ko 키 불균형**: 패치별로 키를 추가하다 보니 일부 키는 `en`에만 있거나, `ko` 번역이 비어 있어 fallback으로 영문이 나오는 경우가 있음.

---

## 벤치마킹·표준 동작

- **HTML `lang`**: 페이지 실제 표시 언어와 맞추면 스크린리더·검색엔진·브라우저 힌트로 동작. SPA는 클라이언트에서 `document.documentElement.lang`을 로케일과 동기화하는 방식이 일반적.
- **`translate="no"` (HTML5 글로벌 속성)**:  
  - 해당 요소(및 하위)를 자동 번역에서 제외.  
  - 브랜드명·고유명사·이미 번역된 앱 UI에 사용.  
  - Google Translate 등이 이 속성을 존중하도록 되어 있음 (일부 경로에서는 미동작 사례도 있음).
- **전체 앱이 자체 i18n(EN/KO 토글)을 쓰는 경우**:  
  - `lang`을 선택된 로케일(en/ko)과 동기화하고,  
  - **`<body translate="no">`** 로 전체를 “번역 제외”하면 브라우저 자동 번역과의 이중 번역을 막을 수 있음.

---

## 개선 옵션

### 옵션 A: 브라우저 자동 번역 방지 (빠른 적용, 기능 영향 없음)

**내용**

- **`<html lang>` 동기화**: 클라이언트에서 선택된 로케일(쿠키)에 따라 `document.documentElement.lang`을 `en` / `ko`로 설정.
- **`<body translate="no">`**: 앱 전체를 브라우저 자동 번역 대상에서 제외해, “Feed → 먹이” 같은 이중 번역 방지.
- **브랜드 “Abstract”**: 로고/앱명에 `translate="no"` 적용 (선택, body에 이미 `translate="no"`가 있으면 중복이지만 명시적).

**장점**

- 구현 범위가 작고, 기존 번역 키·로직을 건드리지 않음.
- 브라우저 번역과의 충돌만 줄여서, **언어적 통일감**은 “우리 앱이 제공하는 문구만 보이게” 하는 방향으로 개선.

**단점**

- KO에서 아직 영문으로 나오는 메뉴/버튼은 그대로임. “완전한 KO 화면”은 옵션 B와 함께 가야 함.

**적용 난이도**: 낮음 (레이아웃 + 작은 클라이언트 컴포넌트).

---

### 옵션 B: i18n 커버리지 확대 (누락·하드코딩 제거)

**내용**

- **키 대칭 점검**: `messages.en`과 `messages.ko`의 키 목록을 비교해, ko에만 없는 키에 한국어 번역 추가.
- **하드코딩 문자열 제거**:  
  - 업로드/설정/온보딩/작품·전시 편집 등에 남아 있는 영문 문자열(라벨, 버튼, placeholder, 에러 메시지)을 `t("...")`로 치환.  
  - 필요 시 `messages.ts`에 새 키 추가 후 en/ko 모두 채움.
- **한국어만 하드코딩된 부분**: 예) “대표 썸네일 저장”, “업로드 실행” 등은 영문 키를 만들고 en/ko 모두 채워 일관되게 관리.

**장점**

- KO 토글 시 실제로 한국어로 보이는 영역이 늘어나 **언어적 통일감**이 크게 좋아짐.
- 이후에도 “새 문자열은 반드시 t() + en/ko 동시 추가” 룰만 지키면 유지보수 가능.

**단점**

- 작업량이 많고, 화면별로 테스트가 필요. 한 번에 하기보다는 **페이지/기능 단위로 나눠서** 적용하는 것을 권장.

**적용 난이도**: 중~높음 (번역 추가 + 코드 수정 + QA).

---

### 옵션 C: A + B + 정기 점검

**내용**

- 먼저 **옵션 A** 적용으로 자동 번역 충돌 제거 및 `lang` 정리.
- 이어서 **옵션 B**를 페이지/모듈 단위로 순차 적용 (우선순위: 공통 네비/버튼 → My/설정 → 업로드·작품·전시).
- 이후 **정기 점검**:  
  - PR 시 “새 사용자 노출 문자열은 반드시 i18n 키 사용” 체크리스트 또는 린트 규칙 도입.  
  - 필요 시 스크립트로 `messages.en` / `messages.ko` 키 목록 비교해 불균형 알림.

**장점**

- 단기: 브라우저 번역 문제 해소.  
- 중장기: KO에서도 거의 전 영역이 한국어로 통일되고, 유지보수 체계가 잡힘.

**단점**

- B 구간에서 리소스(시간/인력) 필요.

---

### 옵션 D: 메타데이터·SEO까지 확장 (선택)

**내용**

- `layout.tsx`의 `metadata.title` / `description`를 로케일별로 분기 (또는 동적 메타).  
- 필요 시 `hreflang` 등으로 언어별 URL/버전 힌트 제공.

**장점**

- 검색 결과·공유 시 언어에 맞는 제목/설명 노출.

**단점**

- App Router에서 로케일별 메타는 쿠키/헤더 기반으로 하려면 구조 설계 필요. 당장 UI 통일감과는 직접 연관이 적음.

---

## 권장 순서

1. **지금 바로**: **옵션 A** 적용  
   - `lang` 동기화 + `body translate="no"` (+ 필요 시 브랜드 `translate="no"`).  
   - 기존 기능/번역 키 변경 없이, 브라우저 자동 번역만 차단.
2. **이어서**: **옵션 B**를 우선순위 높은 화면부터 단계적으로 적용  
   - 예: 공통(nav, 버튼, 알림) → My/설정 → 업로드·작품·전시 편집.  
   - 각 단계마다 “해당 화면 KO 전환 시 모두 한국어”가 되도록 검증.
3. **유지**: **옵션 C**의 정기 점검 룰을 팀 컨벤션으로 두고, 새 문자열은 항상 i18n으로 추가.

이 순서로 가면 “기능에 문제 없이” 언어적 통일감을 단계적으로 개선할 수 있음.
